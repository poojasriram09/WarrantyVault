1. What Is WarrantyVault?

Think of WarrantyVault as a smart digital drawer where you store all your product warranties — and the app reminds you before they expire.

When you buy a phone, laptop, fridge, or any product, it comes with a warranty card or invoice. Most people lose these or forget about warranties until it's too late. WarrantyVault solves this by:

  • Scanning your invoice/receipt using OCR (Optical Character Recognition) — just upload a photo and the app reads the details automatically
  • Tracking expiry dates — you get notified before your warranty expires (7 days, 30 days, etc.)
  • Filing warranty claims — a 3-step wizard guides you through raising a claim
  • Finding service centers — shows nearby authorized service centers in Mumbai
  • Importing invoices from Gmail — scans your email inbox for product purchase emails
  • AI chatbot (WarrantyBot) — answers questions about your warranties using AI

2. The Thinking Process — How We Approached This

Before writing a single line of code, we had to answer: "What problem are we solving? Who uses it? What does it need to do?"

Step 1: Define the Problem

The problem is simple — people lose warranties and forget about them. The solution: a digital locker that automatically reads invoices and tracks everything.

Step 2: Identify the Features (MVP — Minimum Viable Product)

We listed the core features first (things without which the app makes no sense):

  1. User login & accounts (so data is private per user)
  2. Add a product with warranty details
  3. View all warranties with their status (active / expiring / expired)
  4. Get notifications before expiry

Then we added advanced features:

  1. OCR — read invoices automatically
  2. Gmail import — scan emails for invoices
  3. AI chatbot
  4. Claim assistant
  5. Service center finder
  6. Admin panel

Step 3: Choose the Tech Stack

We asked: what technologies can handle this, are free/affordable, and are fast to build with? (See Section 3 for full details.)

Step 4: Design the Database First

We figured out what data needs to be stored before writing any code. This is called schema design. If the database structure is wrong, everything built on top of it breaks.

Step 5: Build in Phases

We didn't try to build everything at once. We built Phase 1 (auth + basic warranty tracking), confirmed it worked, then added Phase 2 (OCR, AI), then Phase 3 (service centers, notifications), and so on.

Key mindset: Always get something working first, then improve it. A simple working app is better than a complex broken one.

6. Database Design — The Foundation

The database is the most critical part. If this is wrong, nothing works. We designed 8 tables in PostgreSQL.

The 8 Tables

Table What It Stores Key Columns
users App user accounts (linked to Firebase) id, firebase_uid, email, display_name, avatar_url
products Physical products (phone, laptop, fridge) id, user_id, product_name, brand, model_number, serial_number, category, purchase_date, purchase_price
warranties Warranty records per product id, product_id, user_id, start_date, end_date, warranty_duration_months, warranty_type
warranties_with_status A VIEW (not a real table) — adds computed "active/expiring/expired" status to warranties All warranty columns + computed status field
documents Uploaded invoices/receipts (stored in Cloudinary, link here) id, product_id, file_url, public_id, file_name, mime_type, ocr_raw_text
claims Warranty claim submissions id, warranty_id, user_id, issue_summary, issue_details, claim_status, submitted_at
notifications In-app expiry alert notifications id, user_id, warranty_id, type, message, is_read, channel
service_centers Authorized repair shops id, brand, name, address, city, area, pincode, phone, email, map_url

Why a View for Status?

We can't store "active/expired" as a column because it changes every day (a warranty that's active today might be expired tomorrow). So we use a database VIEW that calculates it on the fly:

-- Each time you query this view, it runs this logic:
SELECT *,
  CASE
    WHEN end_date < CURRENT_DATE THEN 'expired'
    WHEN end_date <= CURRENT_DATE + 30 days THEN 'expiring_soon'
    ELSE 'active'
  END AS status
FROM warranties;

Row Level Security (RLS)

RLS ensures users can only see their own data. Without it, User A could see User B's warranties. Each table has a policy like:

-- User can only access rows where user_id matches their own ID
CREATE POLICY "Users access own data" ON warranties
  FOR ALL USING (user_id = get_current_user_id());

7. Authentication — How Login Works

We use TWO systems together: Firebase for login identity, and Supabase for storing user data. This is called a "dual auth system."

The Problem

Firebase Auth is excellent for handling login (Google OAuth, email/password, password reset). But our data is in Supabase. We need to link the two.

The Solution

  1. User logs in via Firebase (Google OAuth or email/password)
  2. Firebase gives us a uid (unique user ID) like "Un0hSnDHjgMTHT2d9I9xcBG"
  3. Our app calls a Supabase function sync_firebase_user() which creates/updates a row in the users table using that Firebase UID
  4. Supabase returns the user's internal UUID (like "f47ac10b-58cc-...")
  5. All subsequent Supabase queries use this UUID — NOT the Firebase UID

The useAuth Hook

This is a custom React hook that runs when the app loads. It:

  • Listens for Firebase auth state changes (login/logout)
  • On login: syncs the user to Supabase, fires expiry notifications
  • On logout: clears all user data from Zustand stores
  • Fires on tab visibility change (when user returns to the app)

8. How OCR Works — Reading Invoices Automatically

OCR (Optical Character Recognition) is how computers read text from images. Imagine taking a photo of an invoice and the app automatically filling in the product name, price, and date.

Three-Layer OCR Strategy

Layer 1: PDF Text Extraction (Best Accuracy)

If the file is a digital PDF (like an email invoice from Amazon/Flipkart), the text is already embedded in the file. We use pdfjs-dist to extract it directly — no image recognition needed, 100% accurate.

const content = await page.getTextContent();
const text = content.items.map(item => item.str).join(" ");

Layer 2: AI Vision (High Accuracy for Images)

If the file is an image (photo of a paper invoice), we send it to OpenRouter which routes it to Google's Gemini Flash 1.5 AI model. The AI reads the image like a human and returns structured JSON with all fields.

// We ask the AI:
"Extract product name, brand, model number, serial number,
purchase date, price, retailer, warranty months from this invoice.
Return as JSON."

// AI responds:
{
  "productName": "ASUS VivoBook 15",
  "brand": "ASUS",
  "modelNumber": "X515EA",
  "price": 45999,
  "purchaseDate": "2024-11-15",
  ...
}

Layer 3: Tesseract.js (Fallback)

If both above fail, we use Tesseract.js — a JavaScript port of the open-source Tesseract OCR engine. It runs entirely in the browser (no server needed) and uses regex patterns to find dates, prices, and brand names in the extracted text.

9. How the Gmail Import Works

Instead of manually uploading an invoice, users can let the app scan their Gmail inbox for product purchase emails.

Step-by-Step Flow

  1. OAuth Permission: User clicks "Connect Gmail". A Google popup appears asking to allow read access to Gmail. This uses Firebase GoogleAuthProvider with the gmail.readonly scope.
  2. Search Inbox: The app calls the Gmail API to search for emails matching:
    has:attachment (asus OR samsung OR lg OR reliance OR dell) newer_than:2y
  3. List Emails: Found emails are displayed — showing subject, sender, date, and attachment pills (PDF/image files)
  4. Pick Attachment: User clicks an attachment. The app downloads it using the Gmail API.
  5. OCR: The attachment is fed through the OCR pipeline, which extracts all warranty fields automatically.
  6. Review & Save: Auto-filled form appears. User can edit any field and click "Add to Warranties".

Why Firebase for Gmail OAuth?

Firebase's GoogleAuthProvider can request extra Google API scopes (like Gmail) during login. This is the most straightforward way to get a Gmail API access token in a React app without setting up a separate backend OAuth flow.

10. The Service Layer Pattern

All database operations are in separate "service" files. UI components never talk to the database directly. This is like having a waiter (service) between you (UI) and the kitchen (database).

Why This Matters

If you ever change the database (say, from Supabase to Firebase Firestore), you only need to update the service files — not every component that uses data.

// BAD — component talks to DB directly:
function WarrantyCard() {
  const { data } = await supabase.from("warranties").select("*");
  // UI mixed with data logic
}

// GOOD — component uses service:
function WarrantyCard() {
  const warranties = await warrantyService.getAllForUser(userId);
  // Clean separation
}

Key Services

warranty.service.jsFetch, create, update, delete warranties (always uses the view)
product.service.jsAdd product + warranty in one transaction, upload receipt to Cloudinary

11. State Management with Zustand

State is data that changes over time. "Global state" is data that multiple different components need to know about simultaneously.

The Problem Without State Management

Imagine the TopBar (header) needs to show "Welcome, John" and the Sidebar needs to know if John is logged in. Without global state, you'd have to pass John's user data through every component in between — this is called "prop drilling" and it gets messy fast.

Zustand Stores

We have 3 Zustand stores (global state containers):

StoreWhat It HoldsUsed By
authStore Current Firebase user + Supabase user record Every component that needs user info
warrantyStore Array of user's warranties + active filters (search, status, category) MyWarrantiesPage, WarrantyCard, Dashboard
uiStore UI state — sidebar open/closed, active modals Sidebar, modal components
Important gotcha we hit: Zustand drops JavaScript getters when you call set(). So we moved filteredWarranties out of the store and into a useWarranties hook using React's useMemo. This was a subtle bug that caused the warranties list to always appear empty.

13. Key Features Deep Dive

A. Smart Expiry Notifications

We use Supabase's pg_cron extension (a PostgreSQL scheduler) to run a function daily at 9 AM UTC. This function checks all warranties and creates in-app notifications for those expiring in 0, 1, 7, 30, or 90 days.

The bell icon in the top bar shows unread notification count. Clicking it opens a dropdown panel. We use Supabase Realtime so the count updates live without refreshing the page.

B. The Claim Assistant Wizard

A 3-step form that guides users through filing a warranty claim:

  1. Step 1: Select product (only shows active/expiring warranties)
  2. Step 2: Describe the issue (summary + detailed description)
  3. Step 3: Review details, see nearby service centers, official brand warranty links, submit claim

C. WarrantyBot (AI Chat)

A floating chat button appears on every page. When clicked, it opens a chat panel powered by Groq's free API using the llama-3.3-70b-versatile model. The system prompt is pre-loaded with the user's warranty data so the AI can answer questions like "Which of my warranties expire this month?"

D. Admin Panel

Only accessible to the admin email (triocoders9626@gmail.com). Has 5 tabs:

  • Overview: Platform stats + charts (total users, warranties, claims, category breakdown)
  • Users: All registered users with search
  • Warranties: All warranties with status filter
  • Claims: All claims with the claimant's name, email, avatar
  • System: Tech stack status, database schema, environment variable checker

E. Service Centers

Shows authorized repair centers in Mumbai. Instead of a generic "city" search (all of Mumbai is one city), users can filter by specific Mumbai areas: Andheri, Bandra, BKC, Borivali, Dadar, Dharavi, Kandivali, Kurla, Lower Parel, Malad, Powai, Saki Naka, Thane, Vashi.

The map uses React-Leaflet with dark CARTO tiles — no Google Maps API key required.

14. The Development Workflow

How we actually wrote the code day-to-day.

Local Development

cd "D:/Hackhive Hack/warranty-vault"
npm run dev       # Starts at http://localhost:5173

Vite provides Hot Module Replacement (HMR) — when you save a file, the browser updates instantly without a full page reload.

Database Migrations

Every database change is saved as a numbered SQL file in supabase/migrations/. To apply changes, we paste them into the Supabase SQL Editor and run them. This gives us a full history of every database change.

Environment Variables

API keys and secrets are in .env.local. Vite makes them available in code as import.meta.env.VITE_VARIABLE_NAME. They're prefixed with VITE_ so Vite knows to include them in the frontend bundle.

Security note: .env.local should NEVER be committed to Git. It's in .gitignore. Anyone who gets this file gets access to all your services.

15. Challenges We Faced and How We Solved Them

Challenge What Went Wrong How We Fixed It
Firebase + Supabase auth conflict Supabase RLS policies use auth.uid() which only works with Supabase's own JWTs. Firebase tokens aren't recognized. Opened all RLS policies to USING (true) and enforced data isolation at the application layer (.eq("user_id", userId) in every query)
User session not loading dbUser was null because the Supabase user sync was failing silently Rewrote useAuth.js to always call the RPC first, with a direct upsert fallback, and proper error logging
Warranty status always empty Zustand silently drops JavaScript getters when set() is called — filteredWarranties getter was deleted on every state update Moved filteredWarranties out of the store into a useWarranties hook using useMemo
Blank page on warranty card click formatDate() from date-fns threw a RangeError for invalid/null dates Wrapped in try/catch with isNaN() check, added an ErrorBoundary component
Gmail popup blocked Cross-Origin-Opener-Policy header prevented Firebase popup from communicating back Added same-origin-allow-popups header in vite.config.js
OCR only reading date and brand Regex patterns were too narrow, only catching a few fields Added 8 specific regex patterns + fallback to AI vision model for images
Notifications missing warranties SQL checked end_date = CURRENT_DATE + exact_days — a warranty expiring in 15 days matched no window Changed to range checks: end_date BETWEEN day1 AND day7 etc.

17. Summary — The Full Picture

Here's how all the pieces connect when a user uploads an invoice:

  1. User opens app → Firebase Auth checks if logged in → useAuth syncs user to Supabase → authStore stores user data
  2. User goes to Add Product → React Router renders AddProductPage
  3. User uploads invoice image → ReceiptUploader component → ocrService.extractFromImage()
  4. OCR runs → OpenRouter Vision API reads the image → returns JSON with all product fields
  5. Form auto-fills → handleOCRExtracted() sets form state in React
  6. User clicks Save → productService.addProductWithWarranty() runs:
    • INSERT into products table
    • Compute warranty end date
    • INSERT into warranties table
    • Upload image to Cloudinary → INSERT into documents table
  7. User redirected to My Warranties → warrantyService.getAllForUser() fetches from warranties_with_status view → Zustand store updated → React re-renders the list
  8. Next day at 9 AM UTC → Supabase pg_cron runs → checks expiry dates → creates notifications in DB → Supabase Realtime pushes update → notification bell count updates in real time
